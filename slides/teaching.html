Blobs — Git is a key-value store
.git/ stores your file contents as objects. The key is a hash of the content; the value is the content itself.
📄 main.cpp
int main() {
  greet("world");
}
hash the content
→
716aef…
.git/objects — the key-value store
716aef main.cpp's contents blob
ca6e99 secret.cpp's contents blob
8c5c75 test_1.cpp's contents blob
The catch

A blob is just content — it has no filename. Identical content hashes to the same key, so it's stored once. So how does Git know that 716aef is "main.cpp"?

Tree

A tree gives blobs their filenames — it's a directory listing that points to blobs and other trees by hash.

A tree maps names → hashes. Reveal the structure step by step.
Under the hood · objects
Commits

Every commit creates a new tree (a complete snapshot) — and points to its parent. Unchanged sub-trees are reused.

A commit is the full state of the project at a point in time, linked to the one before it.
Under the hood · refs
Branches

Commit hashes aren't friendly. A branch is just a movable pointer to a commit — a name you can read, that moves as you commit.

Commits are immutable. Branches are mutable — each commit moves the current branch forward.
Under the hood · refs
HEAD

If a branch is a pointer to a commit, HEAD is the pointer to where you are — usually the branch you're currently on.

HEAD = your current position. git switch moves it; a new commit moves HEAD and its branch together.

Remotes

Git is distributed: every clone is a full copy of the history. A remote (usually origin) is just another repo you sync with.

clone copies it down · push sends your commits up · pull brings theirs down. We use these for real in later labs.

The basic command vocabulary
One line each — now that you know what's underneath, the commands are just verbs for moving these objects and pointers around.
git initturn a plain folder into a repo (creates .git/).
git branchlist or create branches (cheap pointers to commits).
git clonecopy a remote repo + its full history to your machine.
git checkout / switchmove HEAD to another branch or commit.
git statuswhat's changed, staged, untracked. Run it constantly.
git mergecombine another branch's work into yours.
git addstage a change for the next commit (into the index).
git cherry-pickcopy a single commit onto your current branch.
git commitrecord the staged snapshot as a new commit.
git push / pullsend / receive commits to a remote. Named now, used in later labs.
Note

Most students will drive Git from VS Code's Source Control panel, which wraps stage, commit, branch, push and pull in buttons — the UI abstracts these commands away. Teach the CLI so they understand what those buttons actually do, but reassure them the UI handles the everyday flow; the terminal is for when it doesn't.

04
Inspecting & undoing
Two more commands you'll reach for often — one to find out who wrote a line, one to throw work away. Handle the second with care.
git blame Read-only · safe
Who wrote this line — and when?
  • Annotates every line of a file with the commit, author and date that last touched it.
  • Answers "who do I ask about this?" and "when did this line appear?"
  • Changes nothing — a pure inspection command.
git reset --hard Destructive
Rewind the branch — and discard everything.
  • Moves the current branch pointer to a chosen commit…
  • …and overwrites your working tree to match it.
  • Uncommitted changes are gone — there is no undo.
  • Plain git reset (--mixed) keeps your edits; --hard does not.
⚠️Want to keep your edits? Use --soft / --mixed. Unsure? Use git stash first, this is your safety net.

05
.gitignore — what not to track
Git tracks everything by default. .gitignore is how you tell it what to leave alone.
  • Build output, virtualenvs (.venv), __pycache__, .pytest_cache/, OS/editor junk, local notes like NOTES.local.md.
  • This repo already ignores .pytest_cache/ — that's why running pytest in the You-do won't pollute your focused commit.
  • git status shows untracked files (Git sees them, you haven't added them) separately from ignored files (Git pretends they aren't there).
Saves you from

The classic "why did my commit include .venv/ or .env?" mistake.

In the lab, you'll see these objects with your own eyes.
