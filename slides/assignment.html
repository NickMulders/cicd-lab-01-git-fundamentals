WARM-UP · 1
Trace a commit through the object graph
Follow the slide's commit → tree → blob chain on a real commit you just made.
1
Add a single sentence to sample-app/README.md.
2
Stage & commit it (see command).
3
git cat-file -p HEAD, then read it aloud. Note the tree SHA and the parent SHA.
4
cat-file the tree → this is the root tree, so the README.md here is the repo's own. Find the sample-app sub-tree entry instead → note its SHA.
5
cat-file the sub-tree → find your README.md entry → note its blob SHA.
6
cat-file the blob → confirm it's your file content. The blob has no filename: the name lived in the tree.
7
Now let Git do that whole walk in one command: git ls-tree HEAD sample-app/. The README.md blob SHA should match the one you traced by hand: that matching SHA is the point.
# step 2: make a commit to trace
git add sample-app/README.md
git commit -m "docs: expand sample-app intro"

# steps 3 to 6: walk the graph
git cat-file -p HEAD
git cat-file -p <tree-SHA>       # root tree → sample-app sub-tree
git cat-file -p <sub-tree-SHA>   # → your README.md blob
git cat-file -p <blob-SHA>

# step 7: same walk, one command, the blob SHA should match
git ls-tree HEAD sample-app/
key → value

Each SHA is a key; cat-file -p prints the value. You're reading Git's key-value store by hand.

Stage by hunk with add -p
A messy working tree becomes three small, logical commits.
1
First reset the practice commit: git reset --hard HEAD~1 (that's fine, we're about to do it properly).
2
Branch off: git switch -c we-do-greeting-polish.
3
Get a messy working tree: either run the seed script, or make three unrelated edits in sample-app/ yourself. Confirm with git status.
4
git add -p → stage one logical change, commit it. Repeat ×3.
5
After each commit, git ls-tree -r HEAD sample-app/, and watch which blob SHA changed.
git reset --hard HEAD~1
git switch -c we-do-greeting-polish

./scripts/seed-messy-state.sh   # option A: seed a known messy tree
# option B: make 3 unrelated edits in sample-app/ yourself
git status

git add -p          # stage one hunk
git commit -m "…"      # repeat ×3

# finally: see the 3 clean commits you just built
git log --graph --decorate --oneline -5
add -p keys

When git add -p stops at each hunk, these are the keys it's waiting for: y/n = stage / skip · s = split into smaller hunks · e = hand-edit · q = quit.

Solo · Phase 1 · whodunnit
PHASE 1 · 1-4
"Who on earth wrote this?"
Every line in a file has an author and a commit. git blame tells you exactly who to thank, or who to blame.
1
Open sample-app/app.py and find the suspicious line in farewell(), the one promising it's "definitely safe to ship."
2
Run git blame sample-app/app.py. Each line shows its commit · author · date.
3
Who wrote that line? 👀 Read the author off the blame output. Anyone you recognise…?
4
Read the full story of that line: git log -p -L :farewell:sample-app/app.py. Blame and log together are how you reconstruct why a line exists.
# 2: author + commit for every line
git blame sample-app/app.py

# 4: the whole history of that function's lines
git log -p -L :farewell:sample-app/app.py
Why it matters

blame isn't for pointing fingers. It's how you find the commit (and the reasoning) behind any line, fast.

Phase 2 · solo · branch, merge & rebase
Make the workflows from the slides happen for real

Three parts — A, B, C. Between them you'll reset the branch back to the same starting commit, so every workflow begins from the same clean history. You'll record that starting point as your first move in Part A.

  • Part A: three clean commits with add -p.
  • Part B: a two-parent merge commit, when history has diverged.
  • Part C: a linear rebase.

Phase 2 · Part A
PART A
Three clean commits
One change per commit. Imagine a reviewer skimming them in 30 seconds.
First · baseline

Before you branch, name your starting point: on main, grab the current commit's short SHA and jot it down as BASE. In Parts B and C you'll git reset --hard BASE to return here, so each workflow starts from the same clean history.

1
git switch -c feature/greeting-tweaks from main.
2
Run the seed script → git status shows three changed files.
3
git add -p → produce three separate commits, each with one change and a sensible message.
# record your baseline first
git switch main
git rev-parse --short HEAD  # note this → "BASE"

git switch -c feature/greeting-tweaks
scripts/seed-messy-state.sh
git status                 # 3 files

git add -p                 # one change…
git commit -m "…"         # …×3
add -p keys

When git add -p stops at each hunk, these are the keys it's waiting for: y/n = stage / skip · s = split into smaller hunks · e = hand-edit · q = quit.

Staging discipline

The whole point of -p: three focused commits, not one blob. If you lump them, redo with git reset HEAD~1 --soft.

Phase 2 · Part B
PART B
A real merge commit
When main has moved on, a merge can't fast-forward: it ties both histories together in a two-parent merge commit.
4
From Part A, feature/greeting-tweaks is three commits ahead; main still sits at your README commit.
5
Put one small unrelated commit on main (edit a file the seed didn't touch, e.g. the root README.md). Now the branches have diverged, so a fast-forward is impossible.
6
git merge feature/greeting-tweaks -m "…". Git can't fast-forward, so it records a merge commit. Passing -m supplies its message inline, so no editor pops open.
7
git cat-file -p HEAD → it lists two parents. git log --graph shows the diamond where the branches rejoin.
git switch main
# edit the root README.md, then stage & commit:
git add README.md
git commit -m "chore: unrelated note on main"

git merge feature/greeting-tweaks -m "Merge feature/greeting-tweaks"   # → merge commit
git cat-file -p HEAD                  # two parents!
git log --graph --decorate --oneline --all
8 · Sketch it

Draw the graph in NOTES.local.md and keep it: in Part C you'll fold in the same feature work with a linear history instead.

Phase 2 · Part C
PART C
Linear rebase
The rebase slide, for real: feature commits get new SHAs on top of main.
9
Reset once more: git switch main && git reset --hard BASE. This drops Part B's merge and its note commit. Redo the note commit so main is ahead of the fork point again; without it, the rebase has nothing to do.
10
git switch feature/greeting-tweaks && git rebase main. The three commits land on the new tip, no merge commit. Notice the new SHAs.
11
git switch main && git merge feature/greeting-tweaks. This fast-forwards. Linear history.
12
Compare your final graph to instructor-notes/lab-key.md. Don't peek before you finish.
git switch main && git reset --hard BASE
# redo the note: edit root README.md again, then
git add README.md
git commit -m "chore: unrelated note on main"

git switch feature/greeting-tweaks
git rebase main            # new SHAs!

git switch main
git merge feature/greeting-tweaks   # fast-forward
git log --graph --decorate --oneline --all

Optional stretch
OPTIONAL · 1
Stretch 1: rebase through a conflict
The higher-value one. Force a real conflict, then resolve it. This is the situation the object model was built to help you survive.
1
Diverge again first. After Part C, main and the feature branch point at the same commit, so a rebase would do nothing. On main, run git reset --hard HEAD~3: the three feature commits leave main but stay on feature/greeting-tweaks.
2
On main, add a commit that edits the same spot inside greet() that the feature's docstring commit touches (e.g. a comment just above the message = line). Same-region edits on both branches are what forces a conflict.
3
git switch feature/greeting-tweaks && git rebase main. Git halts on the conflicting commit.
4
Open sample-app/app.py. Find the <<<<<<< / ======= / >>>>>>> markers and resolve by keeping both changes. Delete the markers.
5
git add sample-app/app.py → git rebase --continue. Confirm linear history.
# 1: pull main back behind the feature branch
git switch main
git reset --hard HEAD~3

# 2: make main collide with the feature edit
# edit the same greet() spot, then:
git commit -am "main: add a comment inside greet()"

# 3: rebase halts on the conflict
git switch feature/greeting-tweaks
git rebase main

# 4 to 5: resolve, then continue
git add sample-app/app.py
git rebase --continue
git log --graph --decorate --oneline --all
Escape hatch

Overwhelmed? git rebase --abort puts everything back exactly as it was. Bailing is always safe.

OPTIONAL · 2
Stretch 2: be git diff, by hand
A deeper dive: rebuild what git diff does internally (a tree walk plus a blob comparison) using only the object tools.
1
Pick two adjacent commits, e.g. HEAD and HEAD~1.
2
git ls-tree -r each one. Which files exist in B but not A? For files in both, which blob SHAs differ?
3
For a changed file, git cat-file -p the old blob and the new blob → compare them and spot the changed line(s) by eye.
4
Now run the real git diff <A> <B>. It should match what you found. You just reimplemented diff.
# 2: walk the two trees, compare entries
git ls-tree -r HEAD~1
git ls-tree -r HEAD
# same path + different SHA  → changed file
# path only on one side      → added/removed

# 3: compare the two blobs
git cat-file -p <old-blob-SHA>
git cat-file -p <new-blob-SHA>

# 4: check yourself
git diff HEAD~1 HEAD
The point

git diff is not magic. It's the key-value store plus a tree walk. The mental model from the slides is now something you can operate by hand.

Optional stretch
OPTIONAL · 3
Stretch 3: cherry-pick one commit
Grab a single commit from a branch without merging the whole thing, and watch it land as a brand-new commit.
1
Start clean: git switch main && git reset --hard BASE. Now main is behind your feature work again.
2
Grab one commit's SHA from git log --oneline feature/greeting-tweaks. Did Stretch 1 first? Pick the docs or test commit — the rewritten docstring commit would conflict again.
3
git cherry-pick <SHA>: Git replays just that change onto main as a new commit.
4
Check git log: the new commit has a different SHA than the original (same change, new identity) and feature/greeting-tweaks is untouched.
git switch main && git reset --hard BASE
git log --oneline feature/greeting-tweaks   # pick a SHA

git cherry-pick <SHA>        # copy one commit onto main
git log --oneline -2         # new commit: NEW sha

# other forms, pick what you need:
git cherry-pick <A> <B>      # a hand-picked few
git cherry-pick <A>~3..<A>   # a contiguous range
Copy, not move

cherry-pick doesn't move a commit: it replays the change as a new commit on your branch. The source branch keeps its original. This is how you grab a single fix without merging a whole branch.
